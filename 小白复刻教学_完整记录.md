# Aether VPN 项目功能复刻完整中文教学

> 给下一个想从零复刻类似功能的小白看的完整实战记录 + 避坑指南
> 本文档基于真实会话记录整理，包含两次主要任务的全部过程。

---

## 任务背景

本次开发共处理了两个独立问题：

1. **按钮布局 Bug**：Speedtest 页面 “Start Speed Test” 按钮在某些屏幕尺寸下排版异常。
2. **新功能开发**：在 Appearance 页面 “MORE OPTIONS” 区域实现两个真实的设置项：
   - Reduce Motion（减少非必要动画）
   - Larger Text（增大 UI 文字缩放，提升可读性）

用户最后反馈“刷新了，没用”，最终通过**重启开发服务器 + 清空持久化数据**才彻底解决。

## 第一阶段：修复 Start Speed Test 按钮布局问题

### 1.1 问题发现流程（推荐复刻方法）

```bash
# 1. 先整体看项目结构
列出 src 目录

# 2. 搜索相关关键词
grep "Start Speed Test" 或 "Speedtest"
```

**关键发现**：
- 按钮定义在 `src/pages/Speedtest.tsx` 第 435 行左右。
- 真正的问题不在 Button 组件本身，而在于**父容器布局**。

### 1.2 根因分析

问题代码（修复前）：

```tsx
<div className="flex flex-col sm:flex-row gap-3 w-full max-w-[320px]">
  <Button ... fullWidth ...>Start Speed Test</Button>
  <Button ... fullWidth ...>Comparison</Button>
</div>
```

`fullWidth` 在 Button 组件内部会添加 `w-full` 类。

- 手机竖屏（`flex-col`）：两个按钮各占满 320px，正常。
- 平板/横屏（`sm:flex-row`）：两个 `w-full` 同时要占满容器 → **严重溢出、错位、按钮被裁切**。

这是在水平 flex 行里同时给两个子元素加 `w-full` 的经典错误。

### 1.3 修复方案（极简且正确）

```tsx
// 修复后
<div className="flex flex-col sm:flex-row gap-3 w-full max-w-[320px]">
  <Button onClick=... size="xl" leftIcon=...>   {/* 去掉 fullWidth */}
    Start Speed Test
  </Button>
  <Button variant="secondary" ... >            {/* 去掉 fullWidth */}
    Comparison
  </Button>
</div>
```

**原理**：
- `flex-col` 模式下，flex 子项默认 `align-items: stretch`，会自动占满宽度。
- `flex-row` 模式下，按钮使用自身内容宽度 + `gap`，自然并排且不会溢出。

同时顺手修复了同文件分享弹窗里类似的 `flex gap-3 + fullWidth` 问题（改成 `flex-1`）。

### 1.4 复刻建议

- 遇到布局问题时，**先用浏览器开发者工具审查元素**，看实际生成的 class。
- 记住：`w-full` 和 `flex` 的主轴方向经常打架。
- 推荐写法：行布局用 `flex-1` 或 `grid grid-cols-2`，而不是 `fullWidth`。

## 第二阶段：实现 Reduce Motion + Larger Text（核心教学）

这是本次最复杂、也最有教学价值的部分。

### 2.1 第一步：探索现有代码（永远不要直接开干）

```bash
# 找到 Appearance 页面
grep -r "MORE OPTIONS" src/

# 发现原来就有占位代码！
```

原来在 `src/pages/Appearance.tsx` 里已经写好了两个开关，但点击后永远是：

```tsx
onCheckedChange={() => toast.info('Coming soon in a future update')}
```

**教学点**：很多需求其实已经有 UI 桩（stub），我们要做的只是把功能真正接上。

### 2.2 第二步：设计整体架构

新功能需要跨 4 个文件改动：

| 文件                  | 职责                           | 为什么必须改 |
|-----------------------|--------------------------------|--------------|
| `src/store/vpnStore.ts` | 持久化用户偏好                 | 需要存到 localStorage |
| `src/App.tsx`           | 全局应用 data-* 属性 + MotionConfig | 控制整个 App 行为 |
| `src/index.css`         | 真正的动画禁用 + 文字缩放规则  | 视觉效果来源 |
| `src/pages/Appearance.tsx` | 用户操作界面 + 即时反馈      | 让用户看到效果 |

### 2.3 详细实施步骤（可照抄）

#### 步骤 A：在 Store 中添加状态（最重要的一步）

在 `vpnStore.ts` 里做三件事：

1. **类型定义**（interface VPNState）：
   ```ts
   reduceMotion: boolean
   largerText: boolean
   ```

2. **默认值**（在 create 内部）：
   ```ts
   reduceMotion: false,
   largerText: false,
   ```

3. **Action + 持久化**：
   ```ts
   setReduceMotion: (enabled) => set({ reduceMotion: enabled }),
   setLargerText: (enabled) => set({ largerText: enabled }),
   ```
   并在 `partialize` 里加上：
   ```ts
   reduceMotion: state.reduceMotion,
   largerText: state.largerText,
   ```

**避坑**：新增字段后，老用户的 localStorage 里没有这两个 key，容易导致 `undefined`。一定要做好默认值保护。

#### 步骤 B：在 App.tsx 全局生效

```tsx
// 1. 引入
import { MotionConfig } from 'framer-motion';

// 2. 读取 store
const { theme, reduceMotion = false, largerText = false } = useVPNStore();

// 3. useEffect 设置属性
useEffect(() => {
  const root = document.documentElement;
  root.setAttribute('data-reduce-motion', reduceMotion ? 'true' : 'false');
  root.setAttribute('data-larger-text', largerText ? 'true' : 'false');
}, [reduceMotion, largerText]);

// 4. 包裹 MotionConfig（让所有 framer-motion 动画自动响应）
<MotionConfig reducedMotion={reduceMotion ? "always" : "user"}>
  <div className="min-h-screen ...">
    <Routes>...</Routes>
  </div>
</MotionConfig>
```

#### 步骤 C：编写强力 CSS（决定最终体验）

在 `index.css` 底部加入：

```css
/* Reduce Motion */
@media (prefers-reduced-motion: reduce),
[data-reduce-motion="true"] {
  *, *::before, *::after {
    transition-duration: 0.01ms !important;
    animation-duration: 0.01ms !important;
  }
  .btn, .glass, .orb, [class*="transition"] {
    transition: none !important;
  }
}

/* Larger Text - 推荐用 1.2 倍（20%）让效果明显 */
[data-larger-text="true"] {
  --text-scale: 1.2;
  font-size: calc(16px * var(--text-scale));
}

[data-larger-text="true"] .text-xs,
[data-larger-text="true"] .text-\[10px\] { ... }
[data-larger-text="true"] .text-\[28px\] { ... }
/* 把项目里真正用到的所有字号都覆盖一遍 */
```

**教学点**：只写理论上的 `text-sm` 是远远不够的。一定要用 grep 找出项目里实际使用的 `text-[28px]`、`text-5xl` 等 arbitrary 值。

#### 步骤 D：在 Appearance 页面接线 + 做“即时可见”预览

这是用户最后反馈“没效果”的最重要解决方案：

```tsx
<GlassCard ...>
  <div className="flex items-center justify-between gap-4">
    <div>
      <div>Reduce Motion</div>
      {/* 实时预览区 */}
      <div className="mt-3 flex items-center gap-2">
        <div className={`w-2.5 h-2.5 rounded-full bg-[#3B82F6] ${!reduceMotion ? 'animate-pulse' : ''}`} />
        <span>{reduceMotion ? 'Motion reduced' : 'Animating normally'}</span>
      </div>
    </div>
    <Toggle
      checked={reduceMotion}
      onCheckedChange={(v) => {
        setReduceMotion(v);
        toast.success(...);
      }}
    />
  </div>
</GlassCard>
```

**为什么必须加实时预览？**
- 全局动画变化对很多人不敏感。
- 20% 文字缩放在某些页面也不够明显。
- 加上卡片内的“活体演示”，用户点一下就能立刻看到变化，极大降低“没效果”的挫败感。

### 2.4 最关键的调试经历（“刷新了，没用”）

用户多次反馈没效果后，我们做了以下事情：

1. **增加预览组件**（上面已经做了）。
2. **把缩放从 1.125 提到 1.2**。
3. **全面覆盖 CSS 选择器**。
4. **最重要的一步**：**重启开发服务器 + 清空 localStorage**。

命令示例（我在终端执行的）：

```bash
# 杀掉旧进程
pkill -f vite

# 重新启动（后台运行）
npm run dev
```

用户在浏览器执行：

```js
localStorage.removeItem('aether-vpn-storage');
location.reload();
```

**根本原因**：
- Vite + Tailwind v4 对新增的深层 CSS 规则 HMR 支持有时不稳定。
- zustand persist + 新增字段，老数据结构会导致状态初始化异常。
- 只有完全重启 + 清缓存才能让新字段 + 新 CSS 真正生效。

## 推荐的小白复刻完整 Checklist

1. **接到需求后不要急着写代码**
   - 先 `list_dir` + `grep` 把相关文件找全。
   - 优先读 `Appearance.tsx`、`vpnStore.ts`、`App.tsx`、`index.css`。

2. **涉及持久化 + 全局效果的功能，提前规划 4 个改动点**（Store + App + CSS + UI）。

3. **新加布尔开关时，永远做好默认值保护**：
   ```ts
   const { reduceMotion = false } = useVPNStore();
   ```

4. **效果不明显的功能一定要做“活体预览”**，否则用户会觉得没反应。

5. **遇到“改了代码刷新没效果”时，优先尝试以下操作（按顺序）**：
   - 硬刷新（Ctrl+Shift+R）
   - 清 localStorage 对应 key
   - 重启 `npm run dev`
   - 删除 `node_modules/.vite` 缓存后重启

6. **使用 todo_write 管理多文件改动**（本次 accessibility 功能就用了）。

## 经验总结（给小白的话）

- **布局问题** 大多是 flex/grid + width 类冲突导致的，审查元素比猜快 10 倍。
- **可访问性功能**（Reduce Motion / Larger Text）是真实产品必须做的，技术上主要是“全局属性 + CSS 规则 + MotionConfig”。
- **持久化 + HMR** 是 React 开发里最容易翻车的组合，遇到奇怪问题时先怀疑缓存和旧数据。
- 写代码时多想想“用户点了开关后，如何在 1 秒内确认自己操作成功了？”—— 这就是为什么我们最后加了实时预览。
